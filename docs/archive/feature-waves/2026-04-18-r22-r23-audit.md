## R-22 — Enrichment queue watchdog

Original scenario: 417 `processing` rows sharing one `locked_at`, stuck 31+ minutes. No reaper, no claim cap, LM Studio 400 holds the lock forever.

### Scenario 1 — Per-claim timeout: configurable and enforced

**Verdict: MET**

- `DEFAULT_LOCK_TIMEOUT_MS = 10 * 60 * 1000` (600 s) in `watchdog.py:36`.
- Wired to `config.toml` via `[brain] lock_timeout_secs = 600` (committed in `config/config.default.toml:28`).
- Brain startup converts it: `lock_timeout_ms = lock_timeout_secs * 1000` (verified in `test_init.py:58,75`).
- `reap_stale_locks(conn, lock_timeout_ms=self.lock_timeout_ms)` called in `server.py:908`.
- SQL: `WHERE status = 'processing' AND COALESCE(locked_at, 0) <= ?` (`watchdog.py:109`).
- Note: default is 600 s (10 min), not the 120 s mentioned in the risk-register mitigation. A row locked for the 31-minute incident would still be reaped in the next reaper cycle after startup. The higher default is intentional headroom for slow LM Studio inference.

### Scenario 2 — Failure tagging: distinct from LLM errors

**Verdict: MET**

- Reaped rows receive `error_message = f"stale lock reaped (>={lock_timeout_ms // 1000}s)"` (`watchdog.py:96`).
- For the default threshold this yields `"stale lock reaped (>=600s)"`.
- LLM-failure rows carry the raw exception string from the enrichment loop — `"stale lock"` is not a substring of any LM Studio error shape.
- Test `test_reaper_attaches_error_message` (`test_watchdog.py:129-140`) asserts `"stale lock" in err`.
- Corpus-health reports can distinguish reap-failures from LLM-failures by substring.

### Scenario 3 — Concurrent-claim cap

**Verdict: MET**

- `DEFAULT_MAX_CLAIM_BATCH = 10` (`watchdog.py:37`); configurable via `[brain] max_claim_batch` in config.
- All four enrichment queues respect the cap:
  - `claim_pending_events_by_session(max_claim_batch=...)` — `enrichment.py`
  - `claim_pending_claude_segments(max_claim_batch=...)` — `claude_sessions.py:484`
  - `claim_pending_browser_events(max_claim_batch=...)` — `browser_enrichment.py:49`
  - `claim_pending_workflow_runs(max_claim_batch=...)` — `workflow_enrichment.py:311`
- Tests cover shell (`test_claim_respects_max_claim_batch`), claude (`test_claim_claude_respects_max_claim_batch`), and workflow (`test_claim_workflow_respects_max_claim_batch`) in `test_watchdog.py`.
- The 435-orphan scenario is structurally prevented: a single claim cycle can claim at most 10 rows per queue.

### Scenario 4 — Startup reconciliation

**Verdict: MET**

- `_reaper_task` is an independent `asyncio.create_task(_reaper_loop())` spawned at `start_enrichment()` (`server.py:916`), **decoupled from the enrichment gather loop**.
- The reaper fires every `poll_interval_secs` (5 s default), including the first cycle ~5 s after startup.
- At startup, any `processing` rows older than `lock_timeout_ms` (600 s) are reaped within the first 5 s.
- Because the reaper is independent, it continues to fire even while `asyncio.gather()` is blocked inside `_call_llm_with_retries` — the docstring explicitly states this (`server.py:898-901`).
- The 31-minute locks from the incident would be reaped in the first reaper cycle after brain restart.
- Minor note: there is no immediate pre-claim reap at boot (the first reap waits `poll_interval_secs`). The enrichment loop also sleeps first (`server.py:903-904`), so the reaper and enrichment loop both wake at ~t+5s. Their ordering in that first cycle is nondeterministic; a claim could precede the first reap by milliseconds. In practice this is harmless: freshly claimed rows are not stale, so the reaper won't touch them.

**R-22 overall: all four scenarios MET.**

## R-23 — `events.git_repo` NULL corpus-wide

Original scenario: 6,790 events with `NULL git_repo`. Daemon never populated it. Project filter silently fell back to `cwd` LIKE only.

### Scenario 1 — Resolution logic: remote-URL-based, not path-based

**Verdict: MET**

- `git_repo.rs:24-33`: `derive_git_repo(cwd)` calls `remote_origin_url(cwd)` → `git config --get remote.origin.url` → parses `owner/repo` slug via `parse_owner_repo()`.
- Both `/Users/carpenter/projects/hippo/` and `/Users/carpenter/projects/hippo-postgres/` (different worktrees, same remote `git@github.com:sjcarpenter/hippo.git`) both resolve to `"sjcarpenter/hippo"` — the remote slug is authoritative, not the filesystem path.
- Fallback to `toplevel_basename` only when no remote is configured (`git_repo.rs:33`).
- Local-path remotes (e.g., `file://` or `/home/me/bare.git`) are rejected by `parse_owner_repo()` and fall back to basename, preventing bogus `owner/repo` slugs (`git_repo.rs:123-129`, test at `:275-297`).
- 500 ms subprocess timeout guards against stuck credential helpers or hung filesystems (`git_repo.rs:16`).

### Scenario 2 — Insert-path wiring: non-NULL `git_repo` for new events

**Verdict: MET**

- `commands.rs:275-277`:
  ```rust
  let git_repo = git_repo
      .filter(|s| !s.is_empty())
      .or_else(|| crate::git_repo::derive_git_repo(std::path::Path::new(&cwd)));
  ```
  Caller-supplied value wins; fallback derives from `cwd`.
- `GitState { repo: git_repo, ... }` flows into `ShellEvent.git_state` (`commands.rs:279-288`).
- `storage.rs:444-452`: unpacks `gs.repo.as_deref()` as `git_repo`.
- SQL (`storage.rs:465-468`): `git_repo` is column 13 in the INSERT, bound via `rusqlite::params![..., git_repo, ...]` at position `?13`.
- Test `test_handle_send_event_shell_derives_git_repo_from_cwd` (`commands.rs:827-882`) creates a temp git repo with a GitHub remote and asserts `gs.repo == Some("sjcarpenter/hippo")`.

### Scenario 3 — Backfill for 6,790 historical NULL rows

**Verdict: UNMET**

- No backfill script exists. `brain/scripts/` contains only `migrate-vectors.py`; `scripts/` contains bulk-enrich, ingest-claude, monitor, re-embed, and security scripts — none for `git_repo` backfill.
- The fix is forward-only: events captured after `eccf617` will have `git_repo` populated. The existing 6,790-row NULL corpus remains untouched.
- Impact: `e.cwd LIKE ? OR e.git_repo LIKE ?` in `mcp_queries.py:157,450` continues to have a dead `git_repo` half for all pre-fix events. Project filter recall for historical events is still cwd-substring-only. The evaluation harness note in `evaluation.py:707-709` already documents this but a data fix has not landed.
- **Filing as a new task** (see below).

**R-23 overall: 2 of 3 scenarios MET. Backfill is UNMET.**
